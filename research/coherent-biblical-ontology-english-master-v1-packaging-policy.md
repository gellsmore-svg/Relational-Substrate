# Coherent Biblical Ontology English Master v1 Packaging Policy

## Purpose

This file defines how the English master should be packaged for readers.

The purpose of the packaging policy is to clarify:

- what belongs in the reader-facing book package;
- what remains source-text only;
- what front matter is appropriate;
- what appendix policy best fits the current master;
- how to package the master without turning it back into a research object.

The purpose is not to change the book's argument.

It is:

```text
to define the reader-facing shell around the stable English master
```

## Central Claim

The central claim is:

> The English master is now stable as a source text, but it is still lightly packaged. The right next packaging move is not to enlarge the core book, but to give it a restrained publication shell: basic title and author presentation, limited front matter, explicit appendix boundaries, and a clear distinction between the main eight-chapter argument and any future companion or specialist material. The correct packaging logic is therefore minimal, disciplined, and book-native.

More briefly:

```text
Package the book cleanly; do not turn the package into a second manuscript.
```

## Core Packaging Rule

The strongest current packaging rule is:

```text
keep the shell light and the spine clear
```

That means the packaging should help the reader enter the book without crowding the argument.

## What the Reader-Facing Package Should Include

The current best package should include:

1. title page
2. author attribution
3. table of contents
4. optional brief preface
5. optional short note on Scripture use and scope
6. the eight-chapter main text
7. optional bounded appendices or appendix notice

## What the Reader-Facing Package Should Not Include

The package should not include:

- research-phase notes
- roadmap language
- calibration or score language
- corpus-state summaries
- derivative-policy logic
- editorial self-commentary

Those belong to governance files, not the book package.

The author should remain:


These should now be treated as the stable publication identity of the English master.

## Front Matter Policy

The best current front matter is restrained.

## 1. Preface

Recommended:

```text
short optional preface
```

Purpose:

- state the book's aim;
- clarify that this is a theological-ontological work, not a popular science book;
- signal that the book argues from Scripture through ontology into material, living, personal, and redemptive order.

The preface should be short enough not to delay entry into Chapter 1.

## 2. Scope Note

Recommended:

```text
short optional note on scope
```

Purpose:

- explain that the book is not a full doctrine of the church;
- explain that AI and heavy formalism are not part of the core spine;
- clarify that the main task is ontological coherence, not exhaustive technical proof.

## 3. Scripture Note

Recommended:

```text
short optional Scripture-use note
```

Purpose:

- state the translation policy if needed;
- explain any consistent stylistic handling of quoted Scripture;
- keep the note brief and non-defensive.

## Table of Contents Policy

The table of contents should be simple and accurate.

It should include:

- book title
- author
- chapters 1-8
- any appendices if present

It should not include:

- stray compilation artefacts;
- working-version labels;
- research descriptors.

## Appendix Policy

The best current appendix rule is:

```text
append only what assists the reader without disturbing the main climb
```

At present, the following materials are appendix-suitable:

- AI boundary material
- short note on technical formalism, if desired
- glossary of core terms

The following materials are not suitable for immediate appendix insertion unless strongly needed:

- deep vorton mathematics
- research score and calibration files
- corpus-state or roadmap material
- long technical stress-test inventories

Those are better handled in companion works.

## AI Appendix Policy

The AI material should not return to the core chapter spine.

If included at all in this book package, it should appear as:

```text
a bounded appendix
```

Its function should be:

- clarificatory;
- comparative;
- secondary to the main machine-life-person arc.

## Formalism Appendix Policy

The technical formalism should not dominate the reader-facing package.

If included, it should be:

- short;
- selective;
- clearly marked as technical background rather than chapter-controlling material.

Otherwise it should remain outside the book as a companion resource.

## Book Apparatus Policy

The best current apparatus level is:

```text
minimal but serious
```

That means:

- no decorative overgrowth;
- no heavy academic framing unless the publishing route requires it;
- enough structure to make the book feel complete.

The current English master is:

```text
source-ready
```

The difference is not ontological.
