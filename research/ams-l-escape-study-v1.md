# AMS L_escape Study v1

## Purpose

This document isolates the most important current geometric descriptor in the interface programme:

- `L_escape`

From:

- `ams-l3-resolution-path-table-v1.md`

the main result was that `L_escape` is now doing real explanatory work in separating the four main `L3` resolution paths:

- successful handoff
- trapped release
- ordered return
- broken multi-loop fragmentation

The problem is that `L_escape` is still being used as an effective but underdefined shorthand.

The aim of this study is therefore:

- to define what escape routes actually are
- to ask what makes them viable or non-viable
- to explain why some elongated stalls complete while others remain trapped, return inward, or fragment

## Core Claim

`L_escape` is not simply “whether there is a way out.”

It is better understood as:

- the degree to which an active stalled path still retains viable completion options consistent with its local organisation

That definition matters because a path may have many nominal directions available while still having poor real escape potential.

So `L_escape` must not be reduced to:

- number of possible paths

It depends instead on:

- whether any of those paths are actually completable
- whether they remain coherent enough to carry the event
- whether local obstruction and boundary geometry allow one of them to dominate

## Working Definition

`L_escape` means:

- the viability of completion routes available to a stalled or partially progressing interface event

This viability depends on:

- structural openness
- coherence continuity
- compatibility of onward route
- obstruction character
- competition from retention or redirection

## What `L_escape` Is Not

It is not:

- simple geometric openness
- simple absence of obstruction
- identical to `B_in`
- identical to `B_out`
- identical to `I_comp`

It overlaps with all of those, but does not collapse into any one of them.

## Relationship to Existing Variables

## `B_out`

`B_out` asks:

- can a whole release an already-organised event into a viable outward transition?

`L_escape` asks:

- once a stalled event exists, are there still viable completion routes left?

So `B_out` is more boundary-release specific, while `L_escape` is a path-viability descriptor inside the evolving stalled event.

## `B_in`

Where admission is relevant, `B_in` asks:

- can the receiving whole admit the event?

`L_escape` asks:

- does the event still possess a route that could reach successful admission at all?

So `L_escape` sits one level closer to the stalled geometry itself.

## `I_comp`

`I_comp` describes:

- the degree of viable joint handoff across an interface

`L_escape` describes:

- whether a given stalled path still has realistic completion options

So `I_comp` is a broader compatibility descriptor, while `L_escape` is a more local, path-sensitive descriptor.

## Five Main Contributors to `L_escape`

The current model suggests five main contributors.

## 1. Path Continuity

The stalled path must still connect forward in a meaningful way.

If local reseating remains active but every continuation path breaks immediately, `L_escape` is low even if the event remains bounded.

This makes `J_trans` one of the strongest contributors to `L_escape`.

## 2. Coherence Preservation

Completion requires not only connection but enough organisational integrity for a route to remain viable.

If coherence weakens too far, the event cannot use its nominal openings effectively.

So `Q_coh` strongly conditions whether escape is real or only apparent.

## 3. Obstruction Character

Not all obstruction lowers escape in the same way.

Some obstruction:

- narrows options but preserves one viable route

Other obstruction:

- destroys continuity across all routes at once

So `L_escape` depends not only on obstruction amount, but on obstruction distribution and structure.

## 4. Directional Competition

A stalled event may have several competing tendencies:

- outward completion
- local retention
- inward return
- fragmentation

Even where some outward route exists, `L_escape` is lower if stronger local or inward attractors dominate.

That means `L_escape` depends partly on what other local resolutions are competing for the event.

## 5. Terminal-Layer Geometry

Some layer shapes preserve viable completion routes better than others.

Examples:

- narrow but continuous channels may preserve escape
- shallow pockets may preserve local retention instead
- broken multi-loop fields reduce escape sharply

So `L_escape` is strongly shaped by the geometry of the active region itself.

## Three Main `L_escape` Regimes

## Regime E1: High Escape Viability

### Description

The stalled or partially progressing event still retains one or more coherent completion routes that can dominate if local mediation continues.

### Typical Signature

- `J_trans`: high or rising
- `Q_coh`: medium/high or high
- `D`: low/medium
- weak domination by retention or return

### Likely Resolution

- successful handoff

### Interpretation

This is the regime where `L3` is more temporary than stable.

## Regime E2: Intermediate Escape Viability

### Description

Some completion routes remain, but they are difficult, unstable, or contested by local retention dynamics.

### Typical Signature

- `J_trans`: medium
- `Q_coh`: medium
- `D`: medium
- local retention still strong

### Likely Resolutions

- trapped release
- unstable partial completion
- conditional handoff

### Interpretation

This is the central `L3` regime.

The event is not closed off completely, but completion does not dominate.

## Regime E3: Low Escape Viability

### Description

Completion routes are weak, broken, or consistently dominated by other local outcomes.

### Typical Signature

- `J_trans`: low
- `Q_coh`: low/medium
- `D`: medium/high or high
- strong retention, return, or fragmentation pressure

### Likely Resolutions

- ordered return
- broken multi-loop fragmentation
- shallow trapped release with no real completion potential

### Interpretation

This is the regime where the event has effectively lost its viable route to completion even if it remains locally active.

## `L_escape` Across the Four `L3` Resolution Paths

## Successful Handoff

`L_escape` must become high enough for one completion route to dominate.

This does not mean the path was easy from the beginning. It means the elongated stall retained enough viable continuity to finish.

## Trapped Release

`L_escape` remains intermediate.

That is one of the strongest current insights in the programme.

The path is not closed, but completion never gains enough dominance to win over local retention.

## Ordered Return

Outward `L_escape` becomes low, but the event may still retain a strong inwardly organised alternative.

This suggests an important refinement:

- outward escape viability
- total directional viability

are not always identical.

For now, `L_escape` should still be read as primarily:

- outward or completion-side viability

## Broken Multi-Loop Fragmentation

`L_escape` becomes low because the event no longer preserves one bounded path strongly enough to carry completion.

Routes may exist geometrically, but no route remains coherently dominant.

## Best Current Benchmarks for `L_escape`

## 1. Glass Release

Best for:

- intermediate `L_escape`
- trapped release versus diffuse release

Why:

- it exposes how completion routes can exist without dominating

## 2. Glass->Crystal Compatibility Stall

Best for:

- `L_escape` in a cross-whole handoff case

Why:

- the event must retain a route not just outward, but into patterned admissibility

## 3. Crystal Return Cases

Best for:

- low outward `L_escape` with preserved organised alternatives

Why:

- they show that low outward escape does not necessarily mean disorder

## What Makes `L_escape` Rise

The current model suggests:

- stronger `J_trans`
- stable or rising `Q_coh`
- reduced disruptive obstruction
- one route becoming more dominant than local looping alternatives
- terminal geometry that preserves continuation instead of shallow retention

## What Makes `L_escape` Fall

The current model suggests:

- falling `J_trans`
- fragmentation of coherence
- obstruction spreading across too many local routes
- terminal geometry that favours pockets or loops over onward continuity
- stronger competition from inward return or local retention

## Important Distinction: Open Space vs Escape

This is one of the strongest conceptual gains in the study.

The event may have:

- open geometric space

without having:

- high `L_escape`

because completion depends on viable organised continuation, not mere emptiness or room.

That is why the descriptor is useful. It is not reducible to physical openness.

## Does `L_escape` Need to Become a Primitive Variable?

Current answer:

- not yet

Reasons:

- it still behaves more like a higher-order geometric descriptor than a primitive substrate variable
- it depends strongly on the existing primitive set
- the current gains are coming from clearer use, not from formalisation

But the pressure is increasing. If the programme keeps relying on `L_escape` at this level, it may eventually need more explicit treatment.

## Current Weak Point

The biggest remaining weakness is this:

- `L_escape` is now clearly useful
- but the model still does not quantify or strongly classify the topology of viable completion routes themselves

That suggests a future need for:

- route-class comparison

rather than an immediate primitive-variable split.

## Confidence

- confidence that `L_escape` is one of the most important current geometric descriptors: `9/10`
- confidence that it is distinct from `B_in`, `B_out`, and `I_comp`: `8/10`
- confidence that it does not yet need to become a primitive variable: `8/10`
- confidence that route viability, not mere openness, is the right interpretation: `9/10`

## Recommendation

Keep `L_escape` as a geometric descriptor for now.

Do not formalise it prematurely.

Instead, deepen the route-level comparison work first.
