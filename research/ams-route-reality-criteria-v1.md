# AMS Route Reality Criteria v1

## Purpose

This document defines the minimum conditions under which a route should count as:

- genuinely completion-directed

rather than:

- merely active
- merely locally coherent
- only apparently open
- or pseudo-progressive

This is the natural next step after:

- `ams-r3-r5-case-matrix-v1.md`

where the strongest discriminator turned out to be:

- did a genuinely completion-directed route ever exist?

That question now needs a more explicit answer.

## Core Claim

A real route is not just:

- motion
- openness
- local coherence
- or even partial advancement

A real route is a path that preserves viable completion logic across enough of the interface process that completion remains genuinely possible rather than merely imaginable.

That means route reality must be judged by:

- viability
- continuity
- directional consistency
- resistance to local looping takeover
- and capacity to remain completion-oriented under progression

## Working Definition

A route counts as genuinely completion-directed when:

- it supports non-trivial progression toward completion
- that progression is not merely local cycling
- continuity remains viable beyond the immediate next step
- no nearer rival local pattern dominates the event completely
- completion remains a real possible outcome of continuing along that path

## What Route Reality Is Not

A route is not automatically real because:

- there is geometric space
- the event is active
- the event is locally coherent
- the path advances briefly
- the path looks open from one side

Those may be necessary in some cases, but they are not sufficient.

## Five Minimum Criteria for Route Reality

## Criterion 1: Forward Continuity

The route must preserve viable continuation beyond immediate first progression.

This is the most basic requirement.

If the route only supports:

- first contact
- or one short step

before guaranteed collapse, then it is not yet a real completion-directed route in the full sense.

### Main Variable Support

- `J_trans`
- secondarily `Q_coh`

### Failure Mode

If forward continuity fails late:

- likely `R3`

If it was never truly there:

- likely `R5`

## Criterion 2: Directional Commitment

The route must preserve a meaningful directional bias toward completion.

This does not mean a perfectly straight or simple path. It means:

- the event is not merely circulating locally
- the path still points the event toward outward release or admissible handoff

### Main Variable Support

- `B_out` or `B_in` depending on case
- `Q_coh`
- `L_escape`

### Failure Mode

If directional commitment is replaced by local cycling:

- likely `R5`

If directional commitment exists but is blocked:

- likely `R3`

## Criterion 3: Completion Viability Under Progression

The route must remain capable of carrying the event through actual progression, not only in static appearance.

This is the criterion that separates:

- apparent route

from:

- viable route

A route may look open, but if it predictably fails as soon as the event actually tries to use it, then it is not robustly real.

### Main Variable Support

- `J_trans`
- `D`
- `L_escape`

### Failure Mode

This is the central failure point of:

- `R3`

## Criterion 4: Dominance Over Local Looping

The route must dominate enough over local reseating alternatives that progression remains the main logic of the event.

This is the clearest criterion separating real route from pseudo-route.

If local cycling is the main event and progression is secondary or illusory, then the route is not genuinely completion-directed.

### Main Variable Support

- relative strength of `J_trans` versus loop retention tendencies
- `L_loop`
- `L_escape`

### Failure Mode

If looping dominates:

- likely `R5`

## Criterion 5: Completion Remains a Live Outcome

The route must preserve completion as a live possibility rather than a purely imagined one.

This is the most synthetic criterion.

It means:

- if the current path continued under the current geometry, completion could still plausibly occur

If completion would require:

- a completely different path
- a totally different geometry
- or a collapse and restart

then the current route is not genuinely real in the relevant sense.

### Main Variable Support

- `L_escape`
- `I_comp` where cross-whole handoff matters
- the whole primitive profile together

### Failure Mode

This is where:

- `R3`
- and `R5`

finally separate most clearly.

In `R3`, completion remains more genuinely live for longer.

In `R5`, completion is often inferred from activity rather than actually preserved.

## Route Reality Test

Current practical diagnostic:

Ask these five questions:

1. Does the path preserve forward continuity beyond first progression?
2. Does it preserve directional commitment toward completion?
3. Does it remain viable under actual progression, not just in appearance?
4. Does it dominate over local looping rather than being produced by it?
5. Does completion remain a live outcome if the event stays on this path?

If the answer is broadly yes across all five:

- the route is genuinely completion-directed

If several answers are weak or no:

- the route is apparent, degraded, or pseudo-route-like

## Route Reality Grades

To make the criteria usable, the programme now needs a rough grading scheme.

## Grade A: Strongly Real Route

The route satisfies all five criteria strongly.

Typical profile:

- high `J_trans`
- medium/high or high `Q_coh`
- low/medium `D`
- high `L_escape`

Typical relation:

- `R1`

## Grade B: Weak but Real Route

The route remains genuinely completion-directed, but under strain.

Typical profile:

- medium `J_trans`
- medium `Q_coh`
- medium `D`
- medium `L_escape`

Typical relation:

- late-stage `R3`
- unstable but real handoff route

## Grade C: Apparent but Unreliable Route

The route shows some completion-like behaviour, but fails one or more core reality criteria often enough that it should not be treated as robustly real.

Typical profile:

- low/medium `J_trans`
- medium or falling `Q_coh`
- medium/high `D`
- low/medium `L_escape`

Typical relation:

- strong `R3`

## Grade D: Pseudo-Route

The event is active and locally coherent, but completion is not a live outcome of the path itself.

Typical profile:

- medium/high local activity
- low/medium `J_trans`
- medium `Q_coh`
- low real `L_escape`
- loop dominance

Typical relation:

- `R5`

## Best Current Application to Benchmark Cases

## Air -> Glass

Typical route reality:

- Grade A / B

Reason:

- viable handoff is genuinely present
- distributed permissive continuation supports real completion

## Glass -> Air

Typical route reality:

- mixed B / C / D depending on terminal layer

Reason:

- clean release can be real
- but many terminal-layer paths are weak, blocked, or pseudo-progressive

## Air -> Polished Metal

Typical route reality:

- strong inward-biased route, weak outward completion route

Reason:

- route reality is real, but not toward outward handoff

This is why route reality and completion-side route reality must be kept distinct conceptually.

## Air -> Oxidised Metal

Typical route reality:

- C / D / fragmentation-edge

Reason:

- routes often appear present but degrade under actual progression

## Glass -> Crystal

Typical route reality:

- B / C / D mixture depending on transition-layer quality

Reason:

- some narrow viable handoff routes are real
- many compatibility routes are only apparently close to admissible completion

## Crystal -> Air

Typical route reality:

- A / B for viable release
- B / C for failed ordered release

Reason:

- the more ordered source whole tends to preserve route reality better even when release fails

## Strongest Current Consequences

## 1. Route Reality Is Now More Than Intuition

This is the main result.

The programme now has explicit minimum conditions for when a route counts as genuinely real.

## 2. `R3` and `R5` Are Now More Firmly Separated

Because:

- `R3` often carries weak but still real route reality
- `R5` often carries activity without robust route reality

## 3. Completion-Side Route Reality Must Be Kept Distinct from Generic Route Activity

This is one of the deepest gains in the whole recent sequence.

It prevents the programme from mistaking:

- intensity of motion

for:

- viability of completion

## Current Weak Point

The current criteria are strong conceptually, but they still need one more layer:

- a direct benchmark scoring pass

That is the next obvious move if the criteria are to become operational rather than only descriptive.

Keep route reality as a criteria-based judgment for now.

Do not formalise it into a primitive variable.

Use the criteria operationally first.
