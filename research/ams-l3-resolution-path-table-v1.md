# AMS L3 Resolution Path Table v1

## Purpose

This document turns the `L3` elongated stall family into a direct side-by-side comparison of its four main resolution paths:

- successful handoff
- trapped release
- ordered return
- broken multi-loop fragmentation

The previous study established that `L3` is the strongest recurring stall family in the current programme. The next task is to stop treating `L3` as a single unresolved middle state and specify what actually separates its possible resolutions.

## Context

From:

- `ams-l3-elongated-stall-family-study-v1.md`

the core claim was:

- `L3` is a bounded but non-shallow interface event in which mediated progression begins but does not immediately complete

That means the real modelling problem is now:

- what determines how `L3` resolves?

| `L3` resolution path | `B_in` / `B_out` emphasis | `J_bound` | `J_trans` | `Q_coh` | `D` | `L_escape` | Dominant result |
|---|---|---:|---:|---:|---:|---:|---|

## Path-by-Path Analysis

## 1. Successful Handoff

### Signature

- relevant release or admission variable becomes high
- `J_bound`: medium/high
- `J_trans`: high
- `Q_coh`: medium/high or high
- `D`: low/medium
- `L_escape`: high

### Interpretation

This is the case where `L3` remains real, but only temporarily.

The elongated stall does not collapse. Instead:

- mediated continuity strengthens enough
- escape routes become viable enough
- local obstruction ceases to dominate

so the event completes.

### Main Driver

The clearest variable shift is:

- `J_trans` rising from medium to high

That appears to be the strongest current indicator that an elongated stall will stop being a stall.

### Best Fit

- successful `glass -> crystal` compatibility after partial mediated transition
- successful but delayed glass release completion

## 2. Trapped Release

### Signature

- `B_out`: low/medium
- `J_bound`: medium/high
- `J_trans`: medium
- `Q_coh`: medium
- `D`: medium
- `L_escape`: medium

### Interpretation

This is the path where the event stays in the elongated stall family rather than leaving it.

The event:

- remains organised enough to preserve the elongated path
- progresses enough to avoid shallow face-only trapping
- but never gains enough continuity to complete

So local reseating persists and terminal continuity never wins.

### Main Driver

The clearest feature is:

- `J_trans` is real but not strong enough

This is the classic stalled-middle regime.

### Best Fit

- glass terminal-layer stall
- deeper trapped release within a terminal layer

## 3. Ordered Return

### Signature

- `B_out`: low/medium
- `J_bound`: high
- `J_trans`: low/medium
- `Q_coh`: high
- `D`: low/medium
- `L_escape`: low outward, but stronger inwardly organised alternatives

### Interpretation

This is the most structured non-completion path.

The elongated path does not collapse into trapping or fragmentation. Instead:

- boundary organisation becomes stronger than transition completion
- directional redirection overtakes outward emergence

So the event resolves into return rather than stall.

### Main Driver

The clearest shift is:

- `J_bound` dominates over `J_trans`

while `Q_coh` remains high enough to preserve directional form.

### Best Fit

- crystal-like exits
- strongly ordered boundaries with failed outward conversion

## 4. Broken Multi-Loop Fragmentation

### Signature

- `B_in` / `B_out`: weak
- `J_bound`: low/medium
- `J_trans`: low
- `Q_coh`: low/medium
- `D`: medium/high or high
- `L_escape`: low

### Interpretation

This is not a successful resolution of the elongated path. It is the breakdown of that path.

The event:

- no longer preserves enough local organisation for one bounded elongated route to dominate
- fragments into several weaker reseating regions
- loses its ability to remain a strong `L3` event

### Main Driver

The strongest pathway into fragmentation is:

- falling `Q_coh`
- weakening `J_trans`
- rising disruptive burden

### Best Fit

- degraded terminal layers
- compatibility regions that lose too much mediated continuity to remain one stall path

## Strongest Pairwise Distinctions

## Successful Handoff vs Trapped Release

Main separator:

- `J_trans`
- `L_escape`

Both may begin from a similar elongated path, but handoff requires:

- stronger continuity
- stronger viable completion routes

Trapped release retains:

- enough mediation for elongation
- not enough for completion

## Trapped Release vs Ordered Return

Main separator:

- whether the path remains a stalled release event
- or becomes a directional boundary-response event

The current strongest distinction is:

- trapped release: medium `Q_coh`, medium `J_trans`
- ordered return: high `Q_coh`, high relative `J_bound`, weaker outward escape

## Ordered Return vs Fragmentation

Main separator:

- coherence preservation

Ordered return:

- preserves enough organisation to redirect directionally

Fragmentation:

- loses enough organisation that no one directed resolution dominates

## Trapped Release vs Fragmentation

Main separator:

- whether one elongated path still dominates

Trapped release:

- yes

Fragmentation:

- no

This is one of the most important discriminator gains in the whole current programme.

## Best Current Control Variables

## 1. `J_trans`

This appears to be the main control variable for whether `L3` remains stalled or actually completes.

It is the clearest separator between:

- successful handoff
- trapped release

## 2. `Q_coh`

This appears to be the main control variable for whether the elongated path remains structured enough for:

- ordered return

or weakens enough to drift toward:

- fragmentation

## 3. Relative `J_bound` / `J_trans`

This ratio-like contrast appears more important than either variable in isolation for distinguishing:

- trapped release
- ordered return

Strong boundary-face organisation with weaker through-layer continuity pushes the system toward return rather than completion.

## 4. `L_escape`

This is one of the most useful geometric descriptors in the table.

It helps explain why similar-looking elongated paths do not resolve the same way.

- high `L_escape`: completion more plausible
- medium `L_escape`: trapping more plausible
- low `L_escape`: return or fragmentation more plausible

This is the main result.

The programme can now say:

- what pushes `L3` toward success
- what keeps it stalled
- what redirects it

That is a major gain.

The table supports the earlier claim that:

- one recurring geometry can connect release and compatibility work

without making them identical.

The current variables plus geometric descriptors still appear sufficient for one more stage of modelling.

The gains are still coming from:

- better geometry
- better discriminators

not from multiplying primitives.

## Current Weak Point

The biggest remaining weakness is not the table itself.

It is that:

- `L_escape`

is doing real explanatory work without yet being formally modelled in a more explicit way.

That does not mean it must become a primitive soon, but it does mean the next geometric pass should probably pressure it directly.
