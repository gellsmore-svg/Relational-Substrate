# Glass-to-Crystal Late-Gate-Dominated Subcase v1

## Purpose

This document adds the second planned case under:

- `ams-case-library-expansion-plan-v1.md`

The specific goal is to isolate a clean:

- `glass -> crystal [late-gate dominated]`

subcase.

The broader `glass -> crystal` benchmark already carries:

- selective successful handoff
- mixed frontier competition
- blocked compatibility routes
- pseudo-compatibility activity

That richness is useful, but it also means the internal structure of the compatibility frontier is still under-separated.

This subcase is meant to fix that by giving the blocked-route side of `glass -> crystal` a clearer formal identity.

## Core Character

This is the glass-to-crystal compatibility subcase in which:

- a genuinely completion-directed compatibility route forms
- narrowing toward crystal-style admissibility is real
- late obstruction becomes decisive near completion
- pseudo-progressive pocketing exists only weakly or secondarily

So the main governing layer is:

- `O2`

and the main route logic is:

- `R3`

rather than:

- `O3`
- `R5`
- or a broad mixed `D4` competition

## Main Current Reading

Best current structural profile:

- route logic is genuinely present
- a narrow admissible path is being approached
- failure occurs late in the compatibility process
- local pseudo-route pocketing does not dominate the whole-interface result

This is the cleanest current glass-to-crystal case for:

- blocked real compatibility route
- late-stage admissibility failure
- weak but real route reality that does not complete

## Why This Is Not the Mixed Frontier Subcase

This must be stated explicitly.

It is **not** the mixed frontier glass-to-crystal case because:

- `R5` is not central
- `O3` is not central
- pseudo-progressive pocketing does not compete strongly enough to define the case

Instead:

- route reality remains the main logic
- completion remains a live outcome for much of the event
- failure happens by late route blocking rather than by pocket-dominant pseudo-progression

## Why This Is Not the Selective Success Subcase

It is also **not** the selective success case because:

- the route does not complete
- admissible handoff remains unrealised
- the late transition into crystal-style order fails

So this case sits exactly between:

- selective success
- mixed frontier pseudo-compatibility

and gives the library a cleaner `R3` anchor inside the compatibility family.

## Route Logic

## Dominant Route Class: `R3`

This is the central route logic.

Why:

- the route is real
- it is completion-directed
- it survives long enough to make completion look genuinely possible
- it fails under actual progression near the end

This is the cleanest compatibility-side `R3` case in the whole programme.

## Secondary Route Class: weak `R1`

This is present only as a neighbour condition:

- the route almost becomes a successful narrow viable route

But it does not complete.

That is why the case is useful. It sharpens the boundary between:

- weak real route
- and actual viable route

## Obstruction Logic

## Dominant Obstruction Class: `O2`

This is the core obstruction logic.

Why:

- the route is preserved for a meaningful duration
- progression is genuine
- the failure is late

This is exactly what late-gate obstruction means in the current framework.

## Secondary Obstruction Class: weak `O1`

This appears as the selective background condition:

- one route is being preserved more than others

But this does not dominate the outcome, because completion is not achieved.

It only explains why a narrow real route is present at all.

That is exactly why `G_C` fits.
