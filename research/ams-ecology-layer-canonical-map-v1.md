# AMS Ecology Layer Canonical Map v1

## Purpose

This file defines the canonical structure of the AMS ecology layer.

The recent ecology work has become one of the strongest parts of the corpus, but it now risks:

- duplication;
- unclear file authority;
- overlapping severe-state concepts;
- uncertain reuse paths.

This note solves that by defining:

1. which ecology files are canonical
2. which variables are core
3. which severe-state concepts are central
4. how the ecology layer should now be reused without unnecessary multiplication

## Central Claim

The central claim is:

> The ecology layer is now strong enough to require canon rather than simple accumulation. Its value will rise further if the corpus treats a small set of files, variables, and severe-state concepts as authoritative and uses the others as supporting elaborations rather than parallel competing centres.

More briefly:

```text
The ecology layer now needs a map more than more territory.
```

## Why Canon Is Needed

The ecology layer now includes:

- dual burden modelling;
- machine-to-cell thresholding;
- recovery ecology;
- synchronisation formalism;
- support formalism;
- reserve formalism;
- coupled ecology loop modelling;
- severe-state comparisons;
- mixed spiral modelling;
- hollow-support modelling.

That is strong work.

But once a layer grows to this size, one new problem appears:

```text
good ideas can begin competing for reuse instead of reinforcing one another
```

This file prevents that.

## Which Variables Are Primary vs Secondary

Not all canonical variables are equal in explanatory centrality.

## Primary Variables

The strongest current primary variables are:

```text
R_res
B_sync
C_sup
```

Why:

- reserve explains headroom;
- synchronisation explains why burdens become ecology-owning;
- support explains whether the ecology still helps itself effectively.

These three now do the most discriminating work.

## Secondary but Necessary Variables

The secondary but necessary variables are:

```text
R_cap
F_casc
```

Why:

- they remain essential to system dynamics;
- but they are more interpretable once reserve, synchronisation, and support are already in view.

## Bridging Variables

The bridging variables are:

```text
A_burden
C_burden
V_E
θ_rec
```

Why:

- they connect machines to ecology;
- they connect ecology to the life boundary;
- they support threshold reading.

## Canonical Severe-State Concepts

The ecology layer should now treat the following severe-state concepts as central:

1. acute shock collapse
2. chronic decay path
3. containment-loss cascade
4. hollow-support condition
5. mixed burden spiral

These five are enough.

They are now the canonical severe-state set for the current corpus.

## Which Severe-State Concepts Are Most Important

Within the severe-state set, two concepts are especially important.

## 1. Hollow-Support Condition

Why:

- it blocks naïve readings of visible compensation;
- it distinguishes structural support from effective support;
- it is crucial for life-boundary precision.

## 2. Mixed Burden Spiral

Why:

- it captures the hardest severe state;
- it explains false recovery and self-consuming compensation;
- it is probably the most realistic severe pattern in the current ecology model.

These two should now be treated as the most reusable severe-state concepts.

## Which Files Are Supporting Rather Than Canonical

Not every ecology-related file should be treated as a top-level reuse centre.

The following are best treated as supporting or exploratory rather than canonical:


These are good files.

But they are best used:

- to justify the canonical model;
- to stress-test it;
- not to replace it as the main reference layer.

## What Not to Do Next

The ecology layer is now strong enough that weak reuse patterns matter.

Avoid:

- creating more severe-state labels unless they clearly outperform the existing five;
- writing new ecology files that redefine the canonical variables without explicit need;
- bypassing `life boundary synthesis v3` when doing upward reuse;
- treating exploratory scenario files as equal to the core model files.

This discipline matters more now than more local ingenuity.
